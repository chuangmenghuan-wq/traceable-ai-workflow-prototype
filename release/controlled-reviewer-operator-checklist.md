# Controlled Reviewer Operator Checklist

**Status:** Public-safe operator checklist  
**Task ID:** LEAD-FLOW-001U  
**Purpose:** quick checklist before answering reviewer follow-up questions

Use this checklist before a controlled reviewer call, demo conversation, or written follow-up.

It is intentionally short. It is not a policy manual, sales guide, pricing guide, launch approval, or implementation playbook.

---

## 1. Before answering

Check:

```text
[ ] Is the reviewer using synthetic or high-level examples only?
[ ] Have I checked the launch readiness checklist?
[ ] Have I checked the review response scope?
[ ] Am I answering from public-safe package files only?
[ ] Is my answer still high-level and non-production?
```

---

## 2. Safe answer sources

Answer only from these public-safe files:

```text
README.md
release/public-safe-package-index.md
release/review-response-scope.md
examples/mini-audit-preview-sample.md
examples/public-safe-review-note-sample.md
docs/chinese-plain-language-overview.md
docs/free-user-objection-response-department.md
docs/public-safe-review-deliverable-preview.md
```

---

## 3. Answer boundaries

Keep answers focused on:

```text
what the prototype is for
what the free preview shows
why synthetic examples are used
why human review matters
what a safe first experiment means
what the sample output demonstrates
what the package currently proves
what the package does not prove yet
```

Do not expand answers into:

```text
a complete operating playbook
a full build guide
a production design
a pricing or sales explanation
a customer qualification process
a real-company implementation plan
```

---

## 4. Safe redirect response

If a question asks for more detail than the public-safe package should provide, use:

```text
This review is meant to explain the public-safe prototype and sample workflow-review structure. I should keep the answer at a high level and use only the reviewed public-safe package files. For details, please refer to the package index and public-safe samples.
```

---

## 5. Stop conditions

Stop and do not continue expanding if:

```text
[ ] The question needs real company data.
[ ] The question asks for a complete method or operating manual.
[ ] The question requires production design.
[ ] The question moves from review into sales, pricing, or qualification.
[ ] The answer would go beyond the package index or public-safe samples.
```

---

## 6. After answering

Check:

```text
[ ] Did I stay within the public-safe files?
[ ] Did I avoid turning the answer into a complete playbook?
[ ] Did I avoid asking for real data?
[ ] Did I avoid making safety, legal, privacy, security, compliance, ROI, or production-readiness guarantees?
[ ] Did I redirect deeper questions back to public-safe samples or the package index?
```
