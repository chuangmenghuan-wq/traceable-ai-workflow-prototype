## 2. Absorbable ideas

### 2.1 Bottom-layer interaction principle

Useful idea:

```text
Before choosing AI tools, understand how humans and AI should interact.
```

Internal interpretation:

```text
AI workflow design should start from interaction mode, not tool choice.
```

Capability seed:

```text
Capability name: Human-AI Interaction Mode Classifier
Purpose: classify whether a workflow should be automation, augmentation, or agency.
Input: workflow idea, expected AI autonomy, data sensitivity, user-facing impact
Output: recommended interaction mode + risk note + required human gate
```

---

### 2.2 Three human × AI modes

Observed modes:

```text
1. Automation: human gives clear instruction, AI executes.
2. Augmentation: human and AI think together; AI fills blind spots.
3. Agency: AI pushes multi-step tasks under higher autonomy.
```

Future Ability Universe interpretation:

| Mode | Good use | Risk | Default boundary |
|---|---|---|---|
| Automation | Clear, repeatable, low-risk task | hidden errors if instruction is wrong | require validation |
| Augmentation | analysis, brainstorming, review, drafting | user may over-trust AI | require human judgment |
| Agency | multi-step execution, tool use, workflow routing | highest operational risk | require strict gate / dry-run first |

Capability seed:

```text
Capability name: AI Mode Risk Router
Purpose: decide whether a workflow is suitable for automation, augmentation, or agency.
Safe first test: classify 5 synthetic workflow examples without executing anything.
Validation: output must include mode, autonomy level, risk reason, and human review point.
```

```text
Do not keep adding AI tools. First reduce to a small stable workflow.
```

Internal interpretation:

```text
Tool Research Department should not chase every tool. It should identify the few tools that support stable, repeatable capabilities.
```

Capability seed:

```text
Capability name: AI Tool Reduction Planner
Purpose: reduce an overloaded tool stack into 2–3 stable workflow roles.
Input: list of tools, user goal, repeated tasks, pain points

```

### 2.5 Context richness principle

Useful idea:

```text
AI output quality depends heavily on useful context.
```

Internal interpretation:

```text
A weak prompt asks AI to produce from nothing. A strong workflow gives AI curated context and a clear output role.
```

Capability seed:

```text
Capability name: Context Richness Checklist
Purpose: check whether a task has enough useful context before asking AI to generate output.
Input: task request + available materials
Output: context score + missing context + safe next question / next action
```

Checklist:

```text
- goal is clear
- audience is clear
- source materials are available
- constraints are known
- examples are available
- decision criteria are known
- risk boundary is clear
- desired output format is clear
- validation method is defined
```

### 2.6 Concept learning three-question method

Useful idea:

When learning an unfamiliar concept, ask:

```text
1. Origin: where did this concept come from, and why was it invented?
2. Relationship: what is it similar to or different from?
3. Use cases: where should it be used, and where should it not be used?
```

Internal interpretation:

```text
Research Department can use this as a default concept-intake pattern.
```

Capability seed:

```text
Capability name: Concept Learning Three-Question Probe
Purpose: turn unknown AI terms, frameworks, or tools into understandable research notes.
Input: concept name + user context
Output: origin, related concepts, suitable scenarios, unsuitable scenarios, capability fit
```

Future Ability Universe extension:

```text
Origin
→ Relationship
→ Use cases
→ Non-use cases
→ Risks
→ Safe first test
→ Capability fit
```

### 2.8 AI assistance, not AI ghostwriting

Useful idea:

```text
AI can organize, optimize, and accelerate, but it should not replace the human's experience and point of view.
```

Internal interpretation:

```text
Future Ability Universe output should preserve human judgment and decision ownership.
```

Capability seed:

```text
Capability name: Human Viewpoint Preservation Gate
Purpose: prevent AI-generated output from becoming generic, over-polished, or detached from the user's actual view.
Input: AI-generated draft + user intent / source viewpoint
Output: pass / revise / blocked + missing viewpoint notes
Validation: output must retain concrete viewpoint, decision logic, and clear boundaries.
```

Blocked outputs:

```text
- generic AI article with no human viewpoint
- overconfident claim not supported by source
- content that sounds like final product marketing when only prototype exists
- invented experience or fake customer proof
```

---

### 2.9 PKM digestion before AI organization

Useful idea:

```text
AI cannot turn ununderstood fragments into real knowledge by itself.
```

Internal interpretation:

```text
Before asking AI to organize knowledge, the system should check whether the user has at least a minimal understanding and intention.
```

Capability seed:

```text
Capability name: Knowledge Digestion Gate
Purpose: decide whether material is ready for AI organization or still needs human understanding first.
Input: raw notes, source materials, user's goal
Output: digest-first / organize-now / research-needed decision
```

Decision rules:

```text
digest-first = user has collected fragments but does not know meaning or purpose
organize-now = goal and categories are clear enough
research-needed = unknown terms or sources need explanation first
```
