## 3. Problem solved

A prototype may be useful internally but unclear externally:

```text
working prototype → unclear problem → reviewer confusion → weak feedback
```

The safer path is:

```text
public-safe prototype
→ problem framing
→ demo walk-through
→ what it does
→ what it does not do
→ why browser-only matters
→ how it maps to Capability Operating System
→ reviewer questions
→ feedback capture
```

## 5. Process

The department process is:

```text
1. Confirm public-safe boundary.
2. Define one-sentence positioning.
3. Define the problem.
4. Define who the demo is for.
5. Prepare a short demo path.
6. Explain what the prototype does.
7. Explain what it intentionally does not do.
8. Connect the demo to Capability Operating System.
9. Add reviewer questions.
10. Add feedback capture format.
11. Run overclaim validation.
12. Require user review before any public use.
```

Use this structure:

```text
WorkflowGuard AI is a browser-only, public-safe prototype for reviewing AI workflow ideas before real data or production systems are involved.
```

Then explain:

```text
It helps preview risk level, safe first test, human review gate, validation checklist, and trace note.
```

Then clarify:

```text
It is not a finished SaaS, not production-ready, not a compliance/security/legal tool, and not meant for real customer data.
```

## 10. Reviewer questions

Use questions such as:

```text
- Is the problem understandable?
- Is the workflow review flow clear?
- Is the browser-only boundary clear?
- Which part feels useful?
- Which part feels confusing?
- What use case would make this easier to understand?
- Does the prototype avoid overclaiming?
```

Avoid asking:

```text
- Can you share your real customer data?
- Can you send internal documents?
- Can we connect to your tools?
- Do you want to buy this now?
- Can we deploy this in production?
```

